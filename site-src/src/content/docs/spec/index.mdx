## Overview

This section contains the human-readable specification for Namnesis. 
The Machine Layer is canonical; this documentation serves as commentary.

:::note[Reading Guide]
- **Normative (†)** sections define requirements
- **Informative (⟂)** sections provide context
- Keywords like **MUST**, **SHALL**, **MAY** follow RFC 2119
:::

---

## Scope

Namnesis specifies:

1. **Capsule Format** — The structure for preserved agent state
2. **Manifest Schema** — Metadata and integrity information
3. **Cryptographic Requirements** — Encryption, signing, and verification
4. **Redaction Policy** — What data is included/excluded
5. **Conformance Levels** — Implementation compliance requirements

---

## Terms

| Term | Definition |
|------|------------|
| **Capsule** | An encrypted, signed archive containing agent state |
| **Manifest** | JSON document describing capsule contents and integrity |
| **Artifact** | A single file or datum within a capsule |
| **Blob** | Encrypted binary data referenced by manifest |
| **Sigil** | Status marker (†, ⟂, ⊕, ⊗, ◇) |
| **Anchor** | Stable URL fragment identifier |

## Conformance

Implementations **MUST** pass the official conformance test suite.
See [Conformance Tests](/conformance/) for details.

:::caution[Normative †]
An implementation claiming Namnesis conformance **MUST**:

1. Produce manifests that validate against the canonical schema
2. Support all required cryptographic algorithms
3. Correctly implement the redaction policy
4. Pass 100% of Level 1 (Witness) conformance tests
:::
